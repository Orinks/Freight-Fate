# Highway Spider — design brief for Fable

**Purpose.** Design a repeatable *methodology* (rules + algorithm) for growing the
Freight Fate freight map along the US interstate / US-highway network — the
"highway spider." Today the map is expanded by hand; this brief asks Fable to
design the systematic version so the map can scale from ~293 cities to a much
larger, well-covered, realistically-routed network without a human wiring every
node. **Fable designs the methodology; Opus implements it as tooling and runs it**
(the same split that built the current map).

This brief front-loads the hard-won judgment from promoting 35 cities by hand so
Fable starts from real constraints, not a blank page.

---

## 1. What Freight Fate is (only what matters here)

Audio-first, accessibility-first trucking sim for blind / low-vision players.
Python 3.12, `uv`. The map is a graph of **cities (nodes)** and **legs (edges)**;
the player dispatches loads between cities and drives legs. Everything is
offline/deterministic — no runtime network calls. All geometry/mileage is baked
at development time from a self-hosted OpenRouteService (truck routing) and OSM
via a self-hosted Overpass.

## 2. Current state (the thing being grown)

- **293 cities, 626 legs.** (Was 258/562 before the last hand-expansion.)
- Data lives in `src/freight_fate/data/world.json` (single source), split to
  `src/freight_fate/data/world_data/` by `tools/index_world.py` (`--check` verifies
  in sync; CI + tests require it).
- **City schema** (`world.py` / `world_models.py`): key is a slug
  `{city}_{st}_{country}` (e.g. `amarillo_tx_us`); fields `spoken_city`, `state`
  (2-letter), `country`, `region` (must equal `classify_region(state, lat, lon)` —
  `regions.py`, 16 canonical regions), `lat`, `lon`, `locations: []` (auto-expands
  to >=2 freight facilities at load — no need to hand-author).
- **Leg schema**: `from`/`to` (city slugs), `miles`, `highway` (shield label),
  `terrain` (`flat|hills|mountain`), and a `corridor` payload (route_points,
  elevation_samples, grade_segments, checkpoints, state_miles, state_crossings,
  interchanges, speed_limits, toll_events) generated by the enrich tooling. Legs
  are undirected; the **unordered endpoint pair must be unique** (no duplicate /
  parallel legs); `10 <= miles <= 800`.
- **Routing**: `World.route_options(from, to, count)` returns alternate routes,
  **deduped by their city sequence** and ranked fastest-first with penalties. So a
  second route between the same pair is only a distinct, pickable option if it runs
  through **different intermediate nodes** — a duplicate straight A→B leg is
  invisible.
- **Hard invariants (tests enforce)**: every city reachable from every other;
  every leg dispatch-complete (route_points/checkpoints/state_miles/elevation/
  grade + a state crossing where endpoints differ); spoken fields never leak slugs,
  2-letter codes, or raw OSM text; region == derived; coords in the CONUS box.

## 3. Tooling that already exists (design to reuse it)

- `tools/add_nodes` (scratch driver): validates + inserts city dicts and bare legs.
- `tools/enrich_routes.py --enrich-all --engine ors` : builds a new leg's corridor
  (geometry, elevation, grades, generated checkpoints, state crossings) from the
  self-hosted ORS truck profile. `--adopt-ors-miles` sets real truck mileage.
  `--add-overpass-pois` (now points at self-hosted Overpass via `OVERPASS_URL`,
  `--rate-limit 0`) adds named truck-stop POIs; `--rural-fuel-fallback` covers
  empty rural legs.
- `place_leg` / `tools/place_checkpoints.py` / `tools/corridor_places.py` : place
  real named town checkpoints on a leg from GeoNames, projected onto the ORS route,
  rejecting towns >2mi off-route.
- `fix_terrain` (mileage-weighted terrain from grade segments — see pitfalls).
- `tools/pick_nodes.py` : read-only GeoNames candidate proposer (ranks US places by
  population, drops existing/within-30mi, prints derived region). A natural seed for
  node selection.
- **Infra**: self-hosted ORS (`ff-ors`, truck routing/mileage/grades) and
  self-hosted Overpass (`ff-overpass`, POIs now, roads later) — both fast, unlimited,
  local. Data sources: GeoNames (population), OSM/Overpass (towns, junctions, stops,
  eventually roads), ORS (mileage/geometry/grades).

## 4. Pitfalls learned promoting 35 cities by hand (the gold)

1. **Nearest-node wiring.** Wire a new node to its 1–2 nearest *existing* nodes on
   the same highway, splitting that corridor. Do NOT wire to the far endpoints of
   the long leg it happened to sit on.
2. **Concurrent-highway trap.** I-20 and I-59 run *concurrent* Birmingham↔Meridian;
   wiring "Birmingham↔Meridian on I-59" duplicated the I-20 path through Tuscaloosa.
   The spider must detect shared roadway and not create redundant parallel legs.
3. **Spur vs gap-filler.** A node <~30mi from a major neighbor (twin cities:
   Odessa/Midland, Kerrville/San Antonio) → a single spur leg (backtrack penalty is
   tiny). A genuine gap-filler on a long corridor → two legs (both directions) so
   through-routing doesn't backtrack 100–300mi.
4. **Alternates need distinct nodes.** To make a second real route pickable (US-97
   vs I-5, US-90 vs I-10, US-95 vs I-84), promote the town *on the alternate* and
   wire the alternate *through it*; never fabricate a duplicate direct leg.
5. **Terrain must be re-derived.** `--enrich-all` sets a new leg's coarse terrain to
   the MAX grade segment, so a few >3% bridge/noise segments wrongly flip flat legs
   to "mountain." Use a mileage-weighted rule (calibrated: mountain if >=18% of
   miles are mountain grade, hills if >=7% mountain or >=48% hills, else flat).
6. **Wrong-road pins.** ORS sometimes picks a faster different road than intended
   (Del Rio→Van Horn defaulted to I-10 not US-90). Detect via checkpoint/mileage
   mismatch; fix with a curated `route_via` pin.
7. **Honest sparsity.** Some western/mountain legs genuinely have no town within 2mi
   and no truck stop — that is realistic, not a bug. Don't force fake nodes.
8. **Region model gaps.** `classify_region` puts all of Idaho in `great_basin`
   though N. Idaho is really PNW — the spider may surface states needing a
   coordinate split rule.

## 5. What Fable should design (the actual ask)

A methodology that, given the current graph + data sources, decides:
- **Node selection.** Which towns become dispatchable nodes? Population floors,
  junction importance (interstate×interstate, interstate×US), freight-generation
  signals, and **target spacing** — and how to avoid both over-densifying (too many
  tiny nodes) and under-covering long gaps. Likely region-aware (dense NE vs empty
  Great Basin).
- **Corridor/adjacency generation.** Formalize §4.1–4.4 into rules: which existing
  nodes a new node wires to; spur vs two-leg; concurrent-road dedup; the
  unique-unordered-pair and reachability constraints; when to add a route-unlock
  alternate.
- **Spider order & termination.** How to traverse (BFS along the interstate graph
  from existing nodes? highway-by-highway?), and coverage/stopping targets per
  highway/region so the map reaches a defined size, not infinity.
- **Route realism check.** How to validate that the resulting graph yields sensible
  `route_options` (real alternates, no absurd detours, fastest-first ranking makes
  sense).
- **Priority/order of corridors** to build first for the most player value.

## 6. Deliverable format

A written methodology doc **plus** language-agnostic pseudocode / decision rules
concrete enough for Opus to implement directly as a `tools/spider_*.py` driver on
top of the existing tooling (§3) and run corridor-by-corridor with per-batch commits
and the test suite as the gate — exactly how the 35 hand nodes were built. Call out
any new data the spider needs (e.g. an interstate-junction graph from Overpass) so
Opus can produce it first.
