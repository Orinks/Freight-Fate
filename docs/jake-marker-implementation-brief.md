# Implementation brief: jake synthesis, jake staging, lane markers

For Phil. Written 2026-07-18 after a design-and-audition session with Norm.
Three deliverables, in dependency order. The audio architecture is settled
and approved by ear; the physics and the wiring are not built.

Prototypes live in `sound-test/` at the repo root (untracked, not ignored).
They import nothing from `freight_fate`, touch no save data, and use a fixed
RNG seed so re-runs are byte-identical. Run any of them with
`uv run --with numpy python sound-test/<name>.py`.

| file | what it does |
| --- | --- |
| `pulse_synth.py` | shared machinery: phase accumulator, resonator banks, cab perspective |
| `jake_v2.py` | per-cylinder jake with staging -- **this is the approved model** |
| `jake_v3.py` | realistic operating points, downshift sequences |
| `rumble_v2.py` | rumble strip on the noise-carrier model |
| `edge_nav.py` | graded edge boundary, stereo |
| `lane_line.py` | marker crossings + seamless jake loop benchmark |

## The one rule that governs all of this

**Anything whose realism lives in exact TIMING is synthesized, never sampled
and never generated by ElevenLabs.** Generative audio renders plausible
texture, not a precise pulse train -- the shipped `vehicle/turn_signal.ogg`
audibly smears into multiple partial clicks per tick, which is what put us
on this path. `tools/generate_sounds.py` is an ElevenLabs API wrapper, not a
DSP synth; it stays correct for one-shot *character* sounds and wrong for
these.

Corollary that decides the architecture: the character of these sounds comes
from suspension, axle and tire-cavity resonances, and **those formants do not
move with speed**. Never pitch-shift a recorded or rendered rumble loop to
change speed -- it drags the formants along and sounds like a toy. Fixed
resonators with a moving excitation rate is the only correct model.

## 1. Jake brake synthesis

Approved by ear ("sounds crazily similar to the actual damn jake"). Port
`sound-test/jake_v2.py` into the game.

### The model

An inline-six releases three times per crank revolution, so the buzz sits at
`f = RPM / 20` Hz -- 60 Hz at 1200 rpm, 105 Hz at 2100. **That is audio rate,
not event rate.** The pops fuse into a pitch and the pitch *is* engine speed.
Do not try to event-schedule it: at 75 Hz the period is 13.3 ms against a
~10.7 ms audio buffer, so scheduling jitter would be ~40% of the period.

What makes it sound real rather than like a synth tone is that the six
cylinders are *not* identical. Model it as six independent pulse trains, one
per cylinder, each firing once per 720 degrees (`RPM/120` Hz), offset a sixth
of a cycle apart in firing order 1-5-3-6-2-4. Each cylinder carries its own
amplitude trim and a slightly detuned exhaust-stack resonance for its runner
length. With all six active they recombine into the `RPM/20` buzz, but now
with a half-order amplitude pattern riding on it -- the lope Norm described
as "grrgrrgrr, not grrrrrrr".

### Runtime strategy: loops, not streaming

Neither audio backend exposes a per-block push stream, and we should not
build one. Instead:

- Generate 6-8 seamless loops at RPM buckets **at startup**. Measured:
  217 ms total for six buckets, ~15x faster than real time. No jake WAV ever
  needs to ship.
- Crossfade between adjacent buckets as RPM moves.
- Rate-shift *within* a bucket for fine detail, but keep buckets narrow
  (~+/-15%) because rate-shifting drags the stack formant.

**Seam requirement:** the lope repeats every TWO revolutions, so a loop must
hold a whole number of four-stroke cycles or it will click at the seam and
scramble the lope. One cycle is `120/RPM` seconds; eight cycles at 1800 rpm
is 533 ms. `lane_line.py::seamless_jake_loop` does this correctly.

### Unverified before you build

`_BassBackend` is the primary backend and I did **not** confirm it accepts
in-memory PCM. The pygame fallback already builds Sounds from numpy arrays
via `sndarray.make_sound` (see `audio.py` around line 339), so that path has
precedent. Check BASS first -- it almost certainly works via a memory stream,
but the whole startup-generation plan rests on it. If it does not, fall back
to rendering the loops to disk on first run and caching them.

## 2. Jake staging in the physics model

Norm asked for this explicitly. Stage is currently not modelled at all.

A three-stage engine brake on an inline-six brakes **2, 4 or 6 cylinders**.

- **Retarding torque** ~ proportional to active cylinders, roughly flat with
  RPM.
- **Retarding power** ~ proportional to (active cylinders x RPM). So the gear
  matters as much as the stage: low gear plus high revs is where the braking
  lives. Peak useful work sits around 1800-2100 rpm.
- This is *why* drivers downshift while jaking, and why the sound everyone
  knows is high-stage, high-rev, punctuated by shift gaps.

### The coupling that makes stage a real decision

Retarding torque goes through the **drive axles only**. Past some fraction of
available grip on snow or ice, the drives break loose -- which is exactly why
real drivers reduce stage or kill the jake in slippery conditions. Wire this
into the existing traction physics and the chain-law work. Without it, stage
is a free button with no downside and should not ship.

Optional, if wanted later: gate stage against no-engine-brake zones.

### Audio consequence

Stage must change the sound's *character*, not just its level -- fewer active
cylinders means gaps in the firing pattern, which is a much stronger lope.
`jake_v2.STAGES` already encodes this.

## 3. Lane-line markers

New feature, no existing asset slot, needs wiring.

### Why this exists

The edge rumble strip could not reproduce the "flurpflurp" Norm remembers,
and the arithmetic says it never will: crossing 3.6 m in ~3 s is ~1.2 m/s
laterally, so a 0.40 m rumble band holds each axle ~330 ms while the axles
are only 90-260 ms apart. Forced overlap, always a lumpy swell.

A raised pavement marker is a different object. The crossing happens at ONE
longitudinal point, so **each axle hits at most one marker**. Front wheel
strikes it, rear wheel strikes the same marker one wheelbase later. In a car
that is two bumps 93 ms apart -- the flurp-flurp. A tractor-trailer turns the
same event into a five-hit roll:

| axle | delay | level at the ear |
| --- | --- | --- |
| steer | 0 ms | 0 dB |
| drive tandems | 211 / 267 ms | -6.6 / -7.8 dB |
| trailer tandems | 474 / 520 ms | -13.5 / -14.1 dB |

(65 mph. Delays scale as 1/speed; levels do not.)

**These are two distinct cues with two distinct meanings.** Markers say "you
crossed the line"; the rumble swell says "you are on the shoulder." Keep them
separate -- a lane change and a drift-off are different mistakes.

### Level

Norm's call: **start louder than physical realism suggests.** This is not a
realism violation. A sighted driver gets a lane change through the whole body;
a blind player has one channel, so amplifying a cue that genuinely exists is
re-encoding information the other senses would have carried. The line is
physical referent, not volume -- amplify real cues freely, never invent cues
with no referent. (We refused an in-lane proximity tick on exactly this
ground.)

### Watch out at low speed

At 35 mph the five hits spread over 938 ms and may read as five separate
events rather than one gesture. If so, fade the trailer hits harder at low
speed rather than letting them straggle.

### Optional spoken "lane change"

Wanted as an option, but speech is too slow to be the real-time cue -- by the
time TTS finishes you have already changed lanes. The marker sound is the
feedback; speech is confirmation. Hang it off the existing verbosity ladder
rather than adding a standalone toggle, and give it a priority rule so it
never steps on a pacenote.

# Addendum, 2026-07-20: the engine, and a rumble correction

Added after a verification pass over the sweep. Everything in this section was
measured against the shipped assets, not inferred. Prototypes:
`sound-test/engine_v1.py` and `sound-test/rumble_v3.py`; rendered WAVs go to
`C:\temp\fftest`.

## 4. The engine loop -- the gap the sweep missed

`docs/sound-shortlist.md` surveys seven needs. None of them is the cruise
engine loop, which is the sound the player hears whenever the truck is moving.
It was never treated as a need, and it is the weakest asset in the game.

### What measurement says

| asset | centroid | <200 Hz | >4 kHz | provenance |
| --- | --- | --- | --- | --- |
| `engine/idle.ogg` | 1109 Hz | 0.41 | **0.07** | Darren Duff, real in-cab |
| `engine/low.ogg` | 7080 Hz | 0.21 | **0.54** | uncredited |
| `engine/mid.ogg` | 7834 Hz | 0.14 | **0.59** | uncredited |
| `engine/high.ogg` | 6748 Hz | 0.23 | **0.52** | uncredited |

A real in-cab diesel is dominated below 1 kHz. Those three are more than half
hiss above 4 kHz, and `CREDITS.md` credits idle, start and shutdown to Duff
while listing low, mid and high nowhere. So the truck starts as a real diesel
and pulls away into something else. **That is the "not loud enough, not
defined" complaint**: the authority of a big diesel lives at the firing
frequency and its first few harmonics, 50-400 Hz, and there is almost nothing
down there. Turning up the mix gain will make it louder without making it a
diesel.

**What is right about them, and must be preserved.** Autocorrelation gives
firing frequencies of exactly 50.0 / 75.0 / 105.0 Hz -- precisely RPM/20 at
1000 / 1500 / 2100, matching the crossfade centres in `audio.py`. The model
was correct. Only the timbre is wrong. Do not "fix" the ladder.

Also concrete: `mid.ogg` has a loop seam discontinuity of 1.85x RMS. It clicks
every 3.6 seconds.

### Do not sample the replacement

The library has no onboard semi cruise loop. Hollywood Edge in particular does
not -- its truck material is air brakes (PE7 92-95), horns and gear grinds,
which is exactly what Duff drew on. The best onboard-diesel beds anywhere in
the 62,280 files are Sound Ideas 2000 Series Track27 *"Diesel Truck, Driving
Along, Interior"*, BBC 05-Transport 01 *"Diesel Lorry -- Interior Startup,
Constant Run, Stop"*, and the 6025 Military Weapons Carrier interior drives.

None of that solves it. You cannot source four recordings at exactly
1000/1500/2100 rpm, so you would pitch-shift one -- dragging the formants,
which is the same trap this brief already names for the rumble. **Use those
three recordings as the reference you tune the resonators against, not as the
shipped asset.**

### The engine is already built

`jake_v2.py` is a diesel engine synth. Same six cylinders on a 720-degree
cycle, same firing order 1-5-3-6-2-4, same per-cylinder trim and skew, same
fixed resonators. The jake releases compressed air; the engine burns fuel.
Different excitation, identical structure. The jake work produced the engine
as a side effect and nobody noticed.

`sound-test/engine_v1.py` is that port. Three layers per cylinder -- combustion
thump into the block (the missing authority), diesel knock from the pressure
rise (load-dependent, this is what makes it a diesel and not a generic
engine), and an intake floor -- then one cab transfer function over the sum,
plus a turbo whine deliberately NOT locked to the firing harmonics.

Measured result: >4 kHz drops from 0.52-0.59 to **0.02-0.03**, centroid lands
at 936-986 Hz against Duff's 1109 Hz. It now matches the one asset in the
family that a real truck driver already accepts.

Two things it buys beyond timbre:

- **Continuous RPM instead of four buckets.** Same startup-generation plan as
  the jake, and the same `_BassBackend` in-memory-PCM question gates both.
- **Exact loop seams.** Each loop holds a whole number of four-stroke cycles,
  with the rate solved backwards from the buffer length so the phase closes to
  under 0.01% of RPM, and circular convolution so resonator tails rejoin at
  the join. Measured seam step is 0.54-0.91x the waveform's own sample-to-
  sample step -- the join is smaller than an ordinary sample transition.

**A measurement warning.** The obvious seam test -- compare head samples to
tail samples -- is wrong for anything containing noise. Broadband noise is
uncorrelated everywhere, so head-vs-tail always looks terrible while tiling it
is perfectly inaudible. I reached for that test first and it reported false
failures on all four loops. What clicks is a *discontinuity*: measure the join
step against the RMS of the signal's own first difference.

### Why the first cut sounded like the jake -- and the fix

Norm, on hearing `engine_v1`: *"still sounds a bit synthesized, and similar to
the jake... I just feel like an actual engine would sound more full."* That is
a correct diagnosis of a modelling error, not an impression.

It sounded like the jake because it **was** the jake. Both used the same
excitation topology -- an instantaneous impulse into a bank of fixed
resonators. For the jake that is physically right: a compression release is
genuinely impulsive, the exhaust valve cracks near TDC and the pressure dumps.
That is *why* the jake is convincing, and it should not change.

Combustion is not impulsive. The gas pushes the piston through ~180 degrees of
crank rotation -- about 20 ms at 1500 rpm. With six cylinders firing every
6.7 ms, three are under pressure at any instant and a real engine never fully
decays between events. An impulse model rings and decays into silence between
firings. **That gap is the missing fullness.**

Three mechanisms want separating, which the first cut collapsed into one:

| layer | excitation | contributes |
| --- | --- | --- |
| torque | sustained pressure over the power stroke | firing frequency and harmonics -- the bottom end |
| ring | the pressure RISE, genuinely impulsive | block and mount modes (shared with the jake) |
| knock | ignition-delay burst | the mid clatter that says "diesel" |

Note the stroke duration must scale with the CYCLE, not wall-clock: at 2100
rpm the same 180 degrees passes in 14 ms. Fixing it in milliseconds makes the
engine progressively hollower as it revs.

**A wrong turn worth recording.** My first implementation convolved the pulse
train with the pressure envelope and fed the result to the block resonators,
normalizing the envelope to unit sum. That makes it a unity-DC smoothing
kernel, which attenuates everything at and above the firing frequency -- the
exact harmonics we were trying to strengthen. It measured backwards: `<200 Hz`
energy *fell* from 0.31 to 0.16 as the stroke lengthened, because the block
modes at 52-390 Hz were being starved rather than sustained. Normalizing to
peak, and splitting the torque layer out as a direct contributor rather than a
filter on the ring, moves it the right way: `<200 Hz` 0.31 -> 0.39 and centroid
991 -> 869 Hz, toward Duff's 0.41 / 1109 Hz.

**The `fullness()` metric did not discriminate** (0.57-0.68 across all
settings, against 0.58 for Duff's real idle). It measures envelope coefficient
of variation, which at 75 Hz is contaminated by the firing waveform itself
rather than isolating the inter-event gap. Treat the spectral shift as the
evidence, not that number, and do not tune against `fullness()`.

### Cycle-to-cycle variation: already solved, do not "fix" it

Correlation between successive firing cycles, swept over f0 (a wrong assumed
period reports a spuriously LOW correlation, so this must be swept):

| | r |
| --- | --- |
| `idle.ogg` (Duff, real) | +0.67 |
| `engine_v1` (synth) | +0.64 |
| `mid.ogg` (shipped) | **+0.996** |

The synth already varies cycle to cycle about as much as a real recording. The
shipped loops are the outlier -- every cycle a near-exact copy, which is what
"electronic" means here and why Norm described the control mix as sounding
like "a turboprop with some fuzz": perfect periodicity is a propeller, and 59%
of the energy above 4 kHz is the fuzz.

**Do not add randomness to fix the electronic quality.** I nearly did, off a
first measurement of r = -0.07 for the real idle that turned out to be a
wrong-period artifact. The remaining synthetic impression comes from two other
places: the loop repeats byte-identically every 0.64 s, and there is no cab
ambience around it. Loop length and an asynchronous bed address those; cycle
jitter does not.

### The bed, and why not the idle

Norm's proposal was to mix the synth with a real in-cab sound, possibly
`idle.ogg`. Right instinct, wrong donor: the idle is a *running engine* with
its own firing frequency, so under a 1500 rpm synth it puts two engines in the
cab at unrelated rates, beating. (`mix_synth_plus_idle_even.wav` demonstrates
it.)

The bed wants to be the NON-periodic part of the cab -- wind, tire, structure.
`vehicle/road.ogg` is already that slot, already 30 s, already plays under the
engine, and is asynchronous to a sub-second engine loop so the composite never
repeats at either period. It is currently ElevenLabs-generated. **Upgrading
that one file to a real in-cab bed is the highest-value asset swap available
and needs no code changes.**

The control mix (`mix_control_shipped_plus_road.wav`) confirms the bed is not
merely masking: the same road bed under the shipped `mid.ogg` still sounds
wrong.

### Formant donors: lifting a real cab off the library

`sound-test/engine_donor_scan.py` scores library candidates as FORMANT donors
(not as playable beds), and `engine_formants.py` fits an all-pole filter by
linear prediction, so each conjugate pole pair becomes one resonance with a
frequency, a bandwidth and a gain. The recording supplies the filter, the synth
supplies the source, and the numbers stay tunable -- Norm's Pianoteq framing,
implemented literally: LPC *is* source/filter separation.

**The scan's first ranking was wrong, in an instructive way.** Its top two
scorers were `hollywood edge/.../21_Interior_Of_Truck_While_Driving.mp3` and
`planes, trains, and automobiles/Interior of truck while driving.mp3` -- the
same source material in two collections. Both measure **r = 0.981**
cycle-to-cycle. They are a tiled loop, not a continuous take. The scorer had
rewarded "very periodic and very steady", which describes an ideal donor and a
loop equally well, so it walked straight into the artifact it existed to avoid.

The consequence is visible in the fit: LPC put a needle-thin pole at 100.9 Hz
(bandwidth 26.5 Hz, gain 1.00) that swamped every other resonance, and the
shaped synth came out at `<200 Hz` = 0.76, centroid 310 Hz. That is a captured
engine harmonic masquerading as a cab resonance -- which would bake one RPM
back into the supposedly fixed filter, the exact failure the whole approach
exists to prevent. A loop penalty is now in the scorer and the "narrow pole"
warning threshold is raised from 25 to 40 Hz.

Ranking by whether the take is actually real, which inverts the original:

| donor | cycle r | verdict |
| --- | --- | --- |
| KMP `Inside hefftruck cabine` | 0.562 | real; clean broad poles, dominant at 261 Hz |
| 6025 Military Weapons Carrier, onboard | 0.718 | **best fit** -- all poles 146-528 Hz wide, no capture, even gains |
| Sony `18 Wheeler In Idle Stop` | 0.881 | borderline; right vehicle class, but no pole below 389 Hz |
| HE / PTA `Interior Of Truck While Driving` | 0.981 | **reject** -- looped, harmonic capture |
| BBC railcar cab (control) | -- | scored low, as a wrong-object control should |

Recommended basis: **6025 Military Weapons Carrier onboard**. Heavy diesel,
onboard perspective, and the only candidate whose poles are all broad enough to
be structural. Through it the synth measures centroid 768 Hz, `<200` 0.30,
`>4k` 0.01. Second choice KMP. Audition `formant_*.wav` against
`engine_1500rpm.wav` (stock filter) before committing.

Caveat that does not go away: LPC cannot distinguish a cab resonance from a
strong engine harmonic. Always read the bandwidth column -- structural
resonances run 70-500 Hz wide, captured harmonics are needle-thin.

## 5. The ignition sequence -- four stages, currently three of them silent

Norm, 2026-07-20: *"you start the engine and wait to 100, there's no engine
starting sounds or anything, you just wait for this god awful sound to
happen."* The one moment the game speaks is the one sound that is wrong, and
the interesting part -- a truck waking up and building air -- is mute.

For a blind player the wait is also uninformative: nothing signals that it is
progressing, or how far along it is.

### The duration is NOT the problem

`sim/vehicle.py` already models this properly:

| constant | value |
| --- | --- |
| `air_cold_start_psi` | 55.0 |
| `air_parking_release_psi` | 100.0 |
| `air_build_idle_psi_per_s` | 4.0 |
| `air_build_fast_psi_per_s` | 7.0 |
| `air_governor_cut_out_psi` | 125.0 |

So 55 -> 100 psi takes **11 seconds at idle**, 6.4 seconds revved. Against
reality that is already *fast*: the CDL standard requires 85 -> 100 psi in 45
seconds or less, and a healthy truck typically manages 1-2 psi/s in that
range, so a real 55 -> 100 is more like 25-45 seconds. The code comment says
the rates are "intentionally compressed," which matches the project rule --
compress rates, never falsify trade-offs.

**Do not lengthen it. Fill it.** Eleven seconds only feels long because
nothing happens. Eleven seconds of a compressor working feels purposeful.

### The four stages and their sources

| stage | sound | source |
| --- | --- | --- |
| 1. crank | starter turning the engine over | `859` (interior, Norm's pick) |
| 2. catch + settle | fires, flares, drops to idle | `859`, then hand to the idle loop |
| 3. build | compressor working, **rising with pressure** | `.854` or `.889` -- slow swells, no valve pop |
| 4. purge at 100 | sharp release, ready to roll | `.855` -- fast bright burst |

The `.854`/`.889` distinction matters and is measurable: both have ~1 s
attacks with **no transient at the front**. Every other air file in the pack
has a valve pop. A pop is a release; a swell is a compressor. That is why the
current ElevenLabs purge sounds wrong for this slot -- it is the wrong gesture
entirely, not just the wrong texture.

### Stage 3 should track pressure, and needs synthesis + sample together

A fixed recording cannot stretch to an arbitrary build time, and the build
time varies with engine speed (4 vs 7 psi/s). So stage 3 wants the same split
as everything else:

- **Periodic layer, synthesized.** A truck air compressor is a piston pump
  driven off the engine, so it pumps at a rate proportional to RPM. That is a
  pulse train and its realism lives in timing -- exactly the synthesis case.
  It also means revving to build faster is *audible*, which is a real
  technique drivers use.
- **Texture layer, sampled.** The broadband hiss from `.854`/`.889`, looped
  or granulated to run for any duration.
- **Pressure mapped to intensity/pitch**, so the player hears 100 approaching
  instead of waiting blind. This is the accessibility win, not the polish.

### Hook point

`audio.py` already has `_begin_engine_start_crossfade` -- the `engine/start`
one-shot plays and crossfades into the idle loop over
`ENGINE_START_CROSSFADE_S`. Stages 1-2 drop straight into that slot. Stage 3
is a new looped bed gated on `air_psi < air_parking_release_psi`, and stage 4
is a one-shot on the crossing.

**Crossfade during settled idle, never mid-transient.** A crossfade is
invisible in steady state and very audible during a flare. And if the synth
runs the measured 859 cab while the recording IS that cab, both sides of the
crossfade are the same truck -- seamless for a physical reason rather than by
careful mixing.

## 6. The idle loop recipe -- and where the synth's boundary is

**Idle, start and shutdown are RECORDINGS. Cruise, engine braking and the jake
are SYNTHESIZED.** That is a principled line, not a compromise, and it was
settled by ear on 2026-07-20.

The synth works at cruise and fails at idle, for a reason worth keeping. At
1500 rpm the firing pulses arrive every 13 ms and **fuse into a tone** -- you
hear a pitch with harmonic structure, which the model produces well. At 647
rpm they are 46 ms apart, well below fusion, so you hear **individual
combustion events**, and each one has to carry the mass of a big cylinder on
its own with no help from fusion. Norm's verdict: "it goes from a full big
engine sound to the sound of a micro engine." The synth's advantage is
continuous RPM and load, which matters when the truck is *working*; at idle
there is no RPM to track and no load to vary, so it gives up nothing to a
recording and has to do its hardest job for no benefit.

### The recipe (source: Splice / Blastwave `SemiTruck_S08IN.859`)

Reproducing this took many failed attempts. In order:

1. **Find the idle by the right metric.** Not steadiness (that finds the truck
   under load), not strongest periodicity (that finds it pulling hard -- a
   working engine is louder and more dominant than an idling one). What
   identifies an idle is **quiet engine plus no motion noise**: low broadband
   above 2 kHz, engine present below 300 Hz. In 859 that is 0-7 s, before the
   air event at 7.2 s.
2. **Skip the settling.** The period is chaotic until 2.7 s and only goes
   properly flat at 3.5 s (sd 0.038 ms). Norm heard this before any
   measurement found it: "it starts out moving quicker, so you need to start
   the loop like a second later."
3. **Measure the true firing rate.** Autocorrelation locks onto the
   SUBHARMONIC here -- every tracker reported ~326 rpm, which is half of the
   real value. The audible subtone is 61.62 ms / 16.2 Hz; true firing is
   **32.36 Hz = 647 rpm**, and one four-stroke cycle is **185.4 ms**.
4. **Notch 586 Hz, Q=45** (13 Hz wide). This is a steady non-engine tone --
   D5 minus 7 cents, which is why it reads as musical -- and it is not locked
   to the engine, so it can never align on a cycle-cut loop. It was the last
   thing audible: "the diesel is perfect" but the tone repeated. Removing it
   is what finished the job.
5. **Correct the drift.** The period still creeps ~0.5% across the passage,
   which accumulates to ~13 ms of phase error over a 21-cycle loop -- a fifth
   of a cycle, easily audible. Split into 0.30 s chunks, resample each by its
   own local ratio to a common 61.62 ms period, crossfade-join at 8 ms.
   Resampling by 0.5% is ~0.09 of a semitone, far too small to drag formants.
6. **Cut on whole four-stroke cycles.** 21 cycles = 3.8821 s. Whole CYCLES,
   not revolutions -- the half-order lope repeats every two turns and an
   arbitrary length scrambles it. 30 ms equal-power crossfade.

Result: `IDLE_FINAL_859.wav`, 3.8821 s, 647 rpm, peak 0.0307.

### Two process lessons

**Never normalise a cut taken from a recording.** `pulse_synth.write_wav`
normalises to 0.85 peak. The idle sits at 0.038 in the source, so writing it
applied ~+27 dB -- which lifts the cabin noise floor with it and makes a quiet
in-cab idle sound like standing outside listening to other trucks. Norm heard
this as a perspective change and was right; the spectra look identical either
way, so no measurement I ran could have caught it. Level is set at playback,
not in the file.

**Granulation is for stochastic textures only.** It solved the gravel
(independent random impacts, so scattering preserves the statistics) and would
destroy an idle (six cylinders in a fixed firing order -- scatter those grains
and the engine stops being an engine). Do not reach for it here.

## 7. Synth "thinness" -- diagnosed, and the post-compact plan

We now have a real interiorized idle at the same RPM as the synth (647 rpm),
so the synth can be differenced against ground truth instead of judged by ear
alone. The deficit spectrum (real / synth, energy-matched 200-1000 Hz):

| band | real vs synth | reading |
| --- | --- | --- |
| 55 Hz | -21 dB | **missing half-order throb** |
| 90 Hz | -29 dB | **missing** |
| 160 Hz | -23 dB | **missing** |
| 1200 Hz | -14 dB | knock-region dip |
| 2400-4800 Hz | +5 to +9 dB | **synth has too much top fizz** |
| 120, 300-850 Hz | ~0 dB | already matches |

So "thin" is specific: **missing low throb (the half-order lope fundamental
and its low harmonics) plus excess high fizz.** That is a few resonators and a
top-end tilt, not a rebuild. The mid-body (300-850 Hz) already matches the real
cab -- the measured-859-formant work landed.

### Plan (for the max-thinking restart)

1. **Cheap first pass, solo:** add half-order content at ~55/90/160 Hz, roll
   off above 2 kHz, re-difference against `IDLE_FINAL.wav`. This may be most of
   the fix and needs no orchestration.
2. **Only if that leaves a wide search:** one ultracode workflow that fans out
   6-8 distinct tunings, each SELF-SCORED against the real-idle deficit
   spectrum to prove it measures right, each rendered to a WAV. Return the best

## Rumble strip: two corrections

**Groove spacing is no longer unverified.** FHWA milled shoulder strips are 12
inch centre-to-centre, 7 inch wide, 0.5 inch deep. 12 in = 0.305 m, so the
0.30 m guess stands and every frequency in the prototypes holds. The duty
cycle is pinned too: 7/12 = 0.58 against the 0.55 in use. Neither number is
free to tune by ear any more. Delete this from "known-unverified".

**The thinness had two causes**, both fixed in `rumble_v3.py`:

1. *Carrier and modulator overlapped.* v2 centred the noise hump at 115 Hz
   while the groove rate is 82 Hz at 55 mph and 104 Hz at 70 -- so at highway
   speed the modulator sat on top of the carrier and above it at 70. AM only
   reads as a crisp buzz when the carrier is well clear of the modulator;
   overlapping gives a wobbling hum. This is also why the transient-heavy
   variant sounded better: the transient layer runs to 1.5 kHz, so turning it
   up was accidentally supplying the missing high carrier content.
2. *The main layer never passed through the structure.* v2 applied the fixed
   resonators to the transient layer only, so the layer carrying most of the
   energy had no axle, suspension or tire cavity in it.

Fixing (2) naively -- routing the modulated carrier through the body bank --
throws away the brightness that was the point, because the bank is four low
modes and behaves like a lowpass. Measured, it dragged the centroid back to
470 Hz however high the carrier was set. **The correct model is two parallel
paths**: structure-borne (groove to tire to axle to frame to seat, arriving
low and thumpy) summed with airborne (contact patch through the glass, top end
intact). v2 had only the airborne path and was thin; body-only is dull.

With the carrier at 900 Hz the centroid moves 828 -> 1773 Hz, and the
transient layer can come back *down* -- at `transient_mix` 1.4 the centroid
falls to 707 Hz, because the transients are low-mid heavy and were only ever a
substitute for the carrier.

**Norm's ruling on shallow-vs-deep is now moot** and the open question can be
closed: keep the truthful 0.58 duty and get identifiability from moving the
carrier *up*, not from deepening the modulation. Brightness reads as "rumble
strip" better than depth does, and it stops the cue fighting the engine bed
for the same low octaves.

**No synthesized gravel** (Norm, 2026-07-20). The library has gravel in depth;
synthesis earns its place only where realism lives in exact timing, which here
is the periodic strip buzz alone. `rumble_v3.py` therefore renders the strip
cue *only*, to ride on a sampled shoulder bed.

**Unresolved architecture.** The jake gets startup-generated loops crossfaded
by RPM; the rumble is wired as a retriggered one-shot `play()`. One baked WAV
freezes the buzz rate -- and speed tracking is the entire reason to synthesize
this rather than record it. It needs the same bucket-and-crossfade treatment
keyed to speed (35/45/55/65/75 mph loops are rendered), or it is a sample with
extra steps.

## What is already wired (do not rebuild)

- `vehicle/rumble_strip` is played from six places in
  `states/driving_updates.py`, already with `pan=self._lane_pan()`, a
  level-scaled `volume=0.25 + rumble * 0.45`, and a matching
  `controller.rumble.rumble_strip(rumble)` haptic call. Panning, intensity
  grading and haptics are plumbed.
- **Caveat:** it is played as a retriggered one-shot via `play()`, not a
  continuous bed. Any replacement asset must be a short seamless loop
  (~0.5 s). Dropping a 4-second bed into that slot stacks into mush and will
  look like a bad sound when it is a length mismatch.
- `src/freight_fate/rumble.py` is a clean 195-line haptics engine with the
  device layer injected (so tests use a recording fake). Motors already map
  to geography: low = left grip, high = right grip.

## Haptics, if you get there

Three gaps, all small, all testable against the existing fake device:

1. `rumble_strip(level)` buzzes both motors -- **no side parameter.** The edge
   grammar depends on side being the instruction. Highest-value change.
2. No marker one-shot. It is a decaying multi-hit envelope, which `_OneShot`
   already supports; the truck's hits land on frames 0/13/16/28/31 at 60 fps,
   so the roll is resolvable and both channels can share timing.
3. Edge textures should grade by structure, not amplitude.

Two classes of pad, do not conflate: ERM pads (Xbox, generic) are intensity-
only with a ~30 Hz practical ceiling, so the jake's 60-105 Hz buzz cannot be
felt but its 12.5 Hz lope can. PS5 DualSense uses voice-coil actuators and is
waveform-driven, so it could in principle take the same synthesized signal --
but SDL's standard `rumble()` does not expose that, and reaching it needs the
DualSense audio endpoint or DS5-specific effect packets. **Keep the
abstraction at effect intent, never at waveform**, so a DS5 backend can
render richly while an ERM pad degrades to envelopes. `rumble.py` already
works this way.

## Known-unverified -- do not treat as fact

- ~~**Groove spacing is a guess (0.30 m).**~~ **RESOLVED 2026-07-20.** FHWA
  milled shoulder strips are 12 in centre-to-centre (0.305 m), 7 in wide, 0.5
  in deep. The guess was right; the duty cycle is pinned at 0.58 as well. See
  the addendum.
- ~~**Cylinder grouping per stage is a guess.**~~ **RESOLVED 2026-07-20, and
  the grouping is not the problem.** Norm reported stage 1 sounding wrong.
  Evenly spacing the two braking cylinders (slots 0 and 3 in firing order
  rather than the adjacent pair at slots 0 and 4) was tried and sounded
  *worse* -- "like a mixer hitting the side of a bowl," which is exactly what
  a perfectly regular ~25 Hz pulse train is: machinery, not an engine.
  The real cause is PITCH FUSION. Two cylinders on a 720-degree cycle at
  1500 rpm is ~25 Hz, which the ear resolves as rhythm rather than tone.
  Stage 3 at 75 Hz fuses into the familiar buzz; stage 1 physically cannot at
  those revs, and no cylinder grouping changes that. It is plausible the model
  is simply CORRECT and a real stage-1 jake is genuinely unimpressive.
  **Recommendation:** keep stage 1 deliberately modest rather than trying to
  make it a small stage 3, or expose only two stages -- many trucks are
  two-stage and most drivers run high anyway. Do not tune stage 1 toward
  drama; there is still no reference recording of one.
- **Nobody has heard a stage-1 jake.** Norm has only ever heard high-stage.
  If no community recording turns up, make the in-game low stage modest and
  plausible rather than dramatic -- do not dramatize a sound we cannot
  confirm. Still true as of 2026-07-20; see the resolved grouping note above.
- **The jake has no reference recording anywhere, and probably will not get
  one.** Absent from Sound Ideas (2026-07-09) and from Blastwave FX
  (2026-07-20). YouTube compilations exist but carry a standard licence, so
  they are out of bounds as a source -- useful for ear comparison only.
  Against one such reference Norm confirmed two things about the model: the
  half-order LOPE is right, and the pitch SLIDES continuously with engine
  speed rather than stepping. Stage 3 is approved by ear and sounds better
  still layered over the engine bed. So the jake is the one component of the
  synthesis stack resting entirely on physics plus one listener -- not a
  reason to distrust it, but the first place to look if it ever sounds off.
- All resonator frequencies and decay times in the prototypes are starting
  guesses. The structure is the claim; the numbers are not.
